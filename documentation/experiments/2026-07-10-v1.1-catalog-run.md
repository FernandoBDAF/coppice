## Defects caught by the run (the method keeps working)

1. **RabbitMQ healthcheck too weak** (EXP-01): `rabbitmq-diagnostics ping`
   passes before the AMQP listener accepts, so `depends_on: service_healthy`
   didn't gate what dependents actually need; api-service fatal-looped 5×
   at every cold boot (masked by `restart: unless-stopped`). Fixed:
   `check_port_connectivity`. Verified: RestartCount=0 across the stack.
2. **publish.py published transient messages** (EXP-09): durable queue ≠
   persistent message; broker restart silently dropped 50/50. Fixed:
   `delivery_mode: 2`. Verified: 50/50 survive.
3. **Publisher-confirm timeouts are invisible** (EXP-03/04): the 5 s confirm
   timeout returns 500 with no error-level log line. Noted in
   CONCEPTUAL_REVIEW §12 — belongs with the v4 observability/retry work.
4. **Cache-outage latency amplification** (EXP-12): documented as a
   discovery finding + CONCEPTUAL_REVIEW §12; candidate fail-fast work
   noted for v4 hardening.

## Calibration notes for future runs

- Email-task 500s at 0.06–0.17% under load/burst are the confirm-timeout
  tail, not a regression — until v4 adds retry/outbox, expect a handful per
  thousand under 50-VU burst.
- EXP-07 wall-clock speedup reads ~2.3× because 3-replica startup (~5 s) is
  inside the window; the slope is the honest number (3.1×).
- After `docker compose stop/start rabbitmq`, the queue table can be empty
  for a few seconds while durable queues recover — poll before judging.
