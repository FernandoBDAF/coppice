# Phase v4 handoff — hardening & scored experiments

**Audience:** the implementation session that finishes v4. Written to be
executed step by step without re-deriving decisions. Read
`documentation/phases/v4-hardening-and-assertions.md` (the *what/why*) once;
this file is the *how*. Architecture choices below are settled — do not
re-litigate them, implement them.

**State you inherit (branch `phase/v4`):** the complete broker topology
already exists as data (`deploy/rabbitmq/definitions.json`, generated by
`scripts/rabbitmq/generate-definitions.py`) plus compilable skeletons with
`TODO(v4)` markers: `operational-workers/internal/common/idempotency/`,
`internal/common/queue/retry.go`, `cmd/loadgen/`,
`api-service/internal/infrastructure/outbox/`,
`auth-service/src/domain/services/JwksService.ts`, `experiments/`
(schema + exp-02 example), `scripts/experiments/run.py`.

**Ground rules**
- After every step: `make verify && make drift-check`, plus the step's own
  acceptance check. Commit per step.
- Changing queue args on an existing broker fails with PRECONDITION_FAILED —
  during development `make nuke` (compose) / recreate rabbitmq STS (kind)
  when topology changes.
- The envelope contract (`graph-worker/shared/contracts/MESSAGE_FORMAT.md`)
  is frozen; you are changing *routing*, never envelope shape.

Suggested order (dependencies): A1 → A2 → A3 → A4 → A5 → A6 → A7 → B1 → B2
→ B3 → B4 → B5. A-track steps are independent enough to reorder slightly,
but A1 must be first (everything declares against it).

## A3 — Retry queues in consumers (ADR-008.1)

Skeleton: `internal/common/queue/retry.go` (tiers + contract comment).

1. Implement `DeathCount` (x-death is `[]any` of `amqp.Table`; sum `count`
   for entries whose `queue` has prefix `<queue>.retry.`). Unit-test with a
   fixture built from a real capture.
2. In `handleDelivery` handler-error path: classify errors — processors
   return `ErrUnretryable` (add to `internal/common/queue/errors.go`;
   validation/unmarshal failures wrap it) vs everything else retryable.
   Retryable: publish body unchanged to `<Exchange>.retry` rk
   `<rk>.retry.<tier>` (persistent, copy headers so x-death survives), then
   ACK. Exhausted or unretryable: publish to `<Exchange>.dlx` rk `<rk>`,
   then ACK. Use one shared channel-safe publisher (the existing
   `queue/publisher.go` has a buggy per-call NotifyPublish — fix it to
   register the confirm listener once, or publish on the consumer channel).
3. graphrag: mirror in Python (`_handle_delivery`): retryable → publish to
   `document-tasks.retry` with tier from x-death, ack; else DLQ. aio-pika
   exposes headers via `message.headers`.
4. Metrics: counter `<type>_retries_total{tier}` + `<type>_dlq_total`.

**Accept:** EXP-40 — flaky-dependency worker (add env
`FAIL_FIRST_N_ATTEMPTS` test hook to the email processor, or use Chaos Mesh
after B3): message visibly waits 5s/30s/2m tiers (RabbitMQ UI), succeeds on
recovery; DLQ only after all tiers; x-death counts match.

## A7 — Sessions, salt drop, roles (ADR-009.2/.6/.7)

One migration wave `auth-service/migrations/002_sessions_roles.sql`:

```sql
CREATE TABLE sessions (
    id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id          UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    refresh_token_id UUID NOT NULL UNIQUE,        -- jti of the CURRENT token
    previous_token_id UUID,                       -- rotation chain, reuse detection
    expires_at       TIMESTAMP NOT NULL,
    created_at       TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    revoked_at       TIMESTAMP
);
CREATE INDEX idx_sessions_user ON sessions(user_id);
ALTER TABLE users DROP COLUMN salt;               -- ADR-009.6
```

1. UserRepository: drop salt handling — `bcrypt.hash(password, 12)` /
   `bcrypt.compare(password, hash)`. ⚠️ Existing hashes were computed over
   `password + salt` — dropping the column invalidates old dev credentials;
   acceptable (lab), note in the migration header. Fix affected tests.
2. Refresh rotation: login creates a session row; `/token/refresh` verifies
   jti matches `refresh_token_id`, rotates (new jti, previous_token_id =
   old); a refresh with a *previous* jti = reuse → revoke session + audit
   log. Logout revokes. Access tokens stay stateless (≤15m).
3. Roles (ADR-009.7): `requiresAuth(['admin'])` on user-management +
   destructive routes; seed admin via migration or env-driven bootstrap
   (`SEED_ADMIN_EMAIL`/`SEED_ADMIN_PASSWORD`, dev defaults in compose).

**Accept:** reuse-detection vitest; admin-only route 403s a normal user;
`\d users` shows no salt column.
